# ML Doc Architecture

## Позиционирование в навигации

- Обновляется существующая страница: `/python/vvedenie-v-python`.
- Sidebar и route остаются без изменений.
- Раздел выступает как вход в Python-ветку перед углублением в `1-core` и ML-пайплайны.

## Каркас документа

### 1) Big Picture

- Показать, зачем Python нужен в ML-стеке и как он связывает data, model training и prod inference.

### 2) Концепция простым языком

- Объяснение через аналогию с frontend:
  - feature schema как API contract;
  - training как build-пайплайн модели;
  - inference как runtime-обработка запроса.

### 3) Инженерная интерпретация

- Явно выделить:
  - Data flow
  - Training flow
  - Inference flow
- Показать, где чаще всего возникают production-риски.

### 4) Минимальный пример кода

- Компактный runnable-кейс на чистом Python:
  - входные признаки;
  - расчет вероятности;
  - применение порога.

### 5) Практический пример

- Ближе к real-world:
  - агрегация событий в признаки;
  - простой train/test цикл;
  - расчет базовых метрик.

### 6) Типичные ошибки

- Формат: «ошибка -> почему это критично -> как исправить».
- Минимум 5 ошибок, включая leakage и train-serving skew.

### 7) Cheat-sheet таблица

- Поля: этап, что делаем, инструменты Python, артефакт, проверка готовности.

### 8) Official docs

- Использовать `<OfficialDocsLinks />`.
- Ссылки только на первоисточники (Python docs, Packaging, pytest, NumPy/Pandas/scikit-learn).

## Анти-дублирование

- Не дублировать глубоко материал `docs/python/1-core/*`.
- Не уводить в детали DL/MLOps deployment (оставить ссылки на профильные страницы).
- Для углубления использовать `<RelatedTopics />` на существующие документы.

## Definition of Ready перед публикацией

- Frontmatter валиден.
- Внутренние ссылки и `RelatedTopics` валидны.
- Оба примера кода воспроизводимы локально.
- Документ читается как инженерное руководство для начинающего в ML frontend-разработчика.
