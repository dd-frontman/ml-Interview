# Documentation Project Analysis

## 1) Структура проекта

- Формат проекта: VitePress-сайт с исходниками в `docs/` и конфигом в `.vitepress/`.
- Контентные разделы первого уровня:
  - `docs/math`
  - `docs/ml`
  - `docs/deep-learning`
  - `docs/mlops`
  - `docs/python`
  - `docs/interview-prep`
- Главная страница: `docs/index.md` (кастомная верстка + ссылки по разделам).
- Навигация:
  - Sidebar генерируется автоматически (`scripts/generate-sidebar.mjs` -> `.vitepress/sidebar.generated.ts`).
  - Связанные темы можно задавать вручную компонентом `<RelatedTopics />`.
  - Автоматические related-темы строятся по тегам из `content-index`.
- Source of truth:
  - markdown-файлы в `docs/**.md`;
  - frontmatter в этих же файлах;
  - метаданные и sidebar генерируются скриптами, а не редактируются вручную.

## 2) Правила оформления и стандарты

- Обязательный frontmatter для всех страниц, кроме `docs/index.md`:
  - `title`, `description`, `tags`, `updatedAt` (`YYYY-MM-DD`).
- Naming conventions:
  - пути/роуты только в `kebab-case`, lowercase latin (`scripts/validate-routes.mjs`);
  - внутренние root-ссылки в формате `/section/page`.
- Качество контента контролируется CI:
  - `validate-content`: frontmatter, запрещенные артефакты, закрытие code fence;
  - `validate-links`: валидность markdown-ссылок, `RelatedTopics`, orphan pages;
  - `validate-routes`: формат роутов и ссылки в sidebar/home;
  - `lint:md`, mobile smoke tests, сборка VitePress.
- Технически доступные UI-блоки:
  - `<OfficialDocsLinks />` (есть компонент, но пока не используется в контенте);
  - `<RelatedTopics />`.

## 3) Текущий стиль документации

- Язык: русский, технический, без академической перегрузки.
- Тон: краткий конспектный.
- Глубина: обзорный уровень, в среднем ~27 строк на страницу.
- Формат: списки и короткие секции; почти нет code-heavy частей.
- Диаграммы: не используются.

## 4) Что уже описано

- Базовые разделы по:
  - математике для ML;
  - supervised/unsupervised ML;
  - основам DL и трансформерам;
  - MLOps;
  - Python toolkit;
  - плану подготовки к интервью.

## 5) Пробелы, дублирование, противоречия

### Пробелы

- Почти отсутствуют воспроизводимые примеры кода.
- Нет единообразных разделов "Типичные ошибки".
- Нет cheat-sheet таблиц в материалах.
- Компонент "Официальная документация" есть, но не используется (`0` страниц).
- Большинство тем покрыты на high-level, без инженерной детализации data/training/inference flow.

### Потенциальное дублирование

- Перекрытие между `ml/obuchenie-s-uchitelem.md` и `mlops/feature-engineering-i-validatsiya.md` в части валидации/утечек.
- Перекрытие между `math/*` и интервью-разделом по базовым проверочным вопросам.

### Противоречия

- Наличие инфраструктуры для richer docs (components, auto-related, строгий CI) vs очень краткий текущий контент.
- Заявленная "комплексная документация" в README vs текущая глубина большинства страниц (конспектный уровень).

## 6) Рекомендации

- Обновлять документы итеративно внутри текущей структуры (без новых верхнеуровневых разделов).
- Для каждой ключевой темы стандартизовать секции:
  - Big Picture;
  - объяснение простым языком;
  - инженерная интерпретация;
  - минимальный код;
  - практический кейс;
  - типичные ошибки;
  - cheat-sheet;
  - official docs.
- Начать с центральной страницы `docs/ml/obuchenie-s-uchitelem.md` как шаблона качества для остальных ML-разделов.
