# ML Research Notes

## Тип ML-задачи

- Базовый сценарий для onboarding: binary classification.
- Пример бизнес-постановки: прогноз конверсии пользователя из `trial` в `paid`.
- Почему выбран этот сценарий:
  - понятен frontend-аудитории через продуктовые события;
  - позволяет объяснить data/training/inference flow на простом кейсе.

## Data flow

1. Источник: продуктовые события (сессии, длительность, саппорт-тикеты, канал привлечения).
2. Агрегация в features per user (например, окно 7/30 дней).
3. Формирование пары `X` (признаки) и `y` (факт конверсии).
4. Разделение на train/test до любых обучаемых шагов.
5. Сериализация признаков/версии схемы для воспроизводимости.

## Training flow

1. Зафиксировать baseline и бизнес-критерий (precision/recall/PR-AUC).
2. Обучить стартовую интерпретируемую модель (например, logistic regression).
3. Зафиксировать параметры и версию модели (веса, threshold, версия признаков).
4. Проверить offline-метрики на test/hold-out.
5. Сохранить артефакты в репозиторий модели/хранилище артефактов.

## Inference flow

1. Получить payload с признаками пользователя.
2. Применить тот же feature contract, что использовался при обучении.
3. Рассчитать `score/proba`.
4. Применить production threshold.
5. Вернуть результат и залогировать метаданные (`model_version`, `feature_version`, latency).

## Используемые библиотеки

- Python stdlib: `dataclasses`, `typing`, `math`, `json`, `pathlib`.
- Для production ML-практики (ссылки в документе):
  - `numpy`
  - `pandas`
  - `scikit-learn`

## Потенциальные риски

- Data leakage (fit/feature-stats на полном датасете до split).
- Train-serving skew (разные правила подготовки признаков offline и online).
- Неверный выбор метрики под бизнес-цель (например, упор в accuracy при дисбалансе классов).
- Отсутствие фиксации версии окружения (`venv`, lock/requirements).
- Ошибки в изменяемых структурах Python (мутация shared-объектов, default mutable args).
- Отсутствие мониторинга после релиза: деградация модели не детектируется вовремя.
